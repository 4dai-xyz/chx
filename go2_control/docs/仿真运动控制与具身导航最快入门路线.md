# 仿真运动控制与具身导航最快入门路线

这份文档对应你的简历项目：

```text
机器人仿真、运动控制与具身导航项目
```

目标不是一下子把 Isaac Sim、MuJoCo、Go2、ORB-SLAM3、DPVO、VGGT、Nav2、VLA 全部学完，而是用最快的路线先做出一个能解释、能演示、能继续扩展的闭环：

```text
仿真场景 -> 传感器数据 -> 位姿估计 -> 轨迹/地图可视化 -> 路径规划 -> 速度控制 -> 机器狗运动
```

你现在本机已经有三块基础：

```text
1. /home/ros/unitree_dev
   Go2 的 MuJoCo / SDK2 / ROS2 开发工作区。

2. /home/ros/isaac_go2/IsaacLab
   Go2 的 Isaac Sim / Isaac Lab 仿真和强化学习工作区。

3. /home/ros/ros2_orbslam3
   ORB-SLAM3 / DPVO / VGGT / KV-Tracker 的视觉定位工作区。

4. /home/ros/unitree_dev/projects/go2_ros2_sdk_ws
   Go2 ROS2 SDK 实机桥接、RViz、SLAM、Nav2、相机/雷达/目标检测学习工作区。

5. /home/ros/unitree_dev/projects/open_source_unitree_rl_gym
   宇树官方强化学习运动控制代码，用于学习 Train -> Play -> Sim2Sim -> Sim2Real 的底层运动控制流程。
```

平台限制：

```text
当前 Ubuntu 是 WSL2。NVIDIA 官方不支持在 WSL2 中运行 Isaac Sim
Python/Kit 图形程序。当前 Isaac 环境可保留用于代码阅读和实验性无窗口计算，
但实时 Isaac 画面需要迁移到原生 Windows、原生 Ubuntu 或远程 Linux GPU。
当前 WSL2 中的实时仿真和控制学习以 MuJoCo 为主。
```

最重要的原则：

```text
先保持三套环境互相独立。
后续只通过 ROS2 topic、TF、Odometry、Path、Image 这些标准接口连接它们。
不要把 Isaac 的 conda 环境、MuJoCo 的 venv、DPVO 的 conda 环境混在一个终端里乱 source。
```

## 1. 项目总架构

你简历里的内容可以拆成 6 层：

| 层级 | 你要做什么 | 当前对应工作区 | 最先能跑的东西 |
|---|---|---|---|
| 仿真层 | 创建机器狗和室内/地形环境 | `unitree_dev`、`isaac_go2` | MuJoCo Go2、Isaac Go2 |
| 控制层 | 给 Go2 发速度、姿态、步态、急停等命令 | `unitree_sdk2`、`unitree_sdk2_python`、`unitree_ros2` | MuJoCo 中发控制命令 |

当前 ORB-SLAM3 wrapper 里已经有这些接口：

| topic | 类型 | 用途 |
|---|---|---|
| `/camera/image_raw` | `sensor_msgs/msg/Image` | 输入图像 |
| `/yolo/dynamic_mask` | `sensor_msgs/msg/Image` | 动态物体 mask |
| `/slam/pose` | `geometry_msgs/msg/PoseStamped` | 相机位姿 |
| `/slam/odom` | `nav_msgs/msg/Odometry` | 相机里程计 |
| `/slam/tracking_state` | `std_msgs/msg/Int32` | 跟踪状态 |
| `/slam/map_points` | `sensor_msgs/msg/PointCloud2` | 地图点 |
| `/slam/debug_image` | `sensor_msgs/msg/Image` | 调试图像 |

你要学会的不是“ORB-SLAM3 名字”，而是这条链：

```text
相机图像 -> 特征点 -> 跟踪 -> 局部建图 -> 回环 -> 位姿 -> ROS2 pose/odom/path
```

关键注意：

```text
单目 SLAM 有尺度不确定性。
直接把单目 /slam/odom 喂给导航系统，可能会出现比例不对的问题。
后续需要用 IMU、腿部里程计、已知相机高度、标定尺度、或者 stereo/RGB-D 来解决尺度。
```

### 第 6 步：把视觉定位接到 ROS2 导航链

这是最关键的“从仿真到具身导航”工程连接点。

标准 ROS2 导航坐标链：

```text
map -> odom -> base_link -> camera_link
```

你现在 ORB-SLAM3 输出接近：

```text
world -> camera
```

所以后续要补一个 bridge，把视觉 SLAM 的相机位姿转换成机器人底盘可用的位姿：

```text
/slam/odom
    -> 坐标系转换
    -> 尺度修正
    -> 相机外参 camera_link -> base_link
    -> /visual_odom 或 /odometry/visual
    -> robot_localization EKF
    -> /odometry/filtered
    -> Nav2 / controller
```

第一版可以先做简单可视化：

```text
1. ORB-SLAM3 输出 /slam/pose。
2. 写一个节点把 PoseStamped 累积成 nav_msgs/Path。
3. 在 RViz 里显示轨迹和地图点。
4. 不控制 Go2，只证明定位链路跑通。
```

第二版再做闭环：

```text
1. Isaac 相机发布 /camera/image_raw。
2. ORB-SLAM3 输出 /slam/odom。
3. 规划器给出目标点。
4. 控制器把目标点转成 /cmd_vel。
5. Go2 high-level bridge 把 /cmd_vel 转成 Unitree SportClient 速度命令。
6. MuJoCo 或 Isaac 中 Go2 根据速度命令运动。
```

### 第 7 步：再接 Nav2 和具身导航

Nav2 是第二阶段。不要一开始就把 Nav2、SLAM、Isaac、Go2 全部同时接，调试会非常痛苦。

推荐顺序：

```text
1. 先学 Nav2 的输入输出：
   map、odom、base_link、cmd_vel、goal_pose、costmap、path。

2. 再在简单机器人或 TurtleBot 示例里跑通 Nav2。

3. 再把 Go2 的速度控制桥接成 /cmd_vel。

4. 再把视觉 SLAM 或仿真 ground truth 作为 odom 输入。

5. 最后才换成 Isaac 室内场景和真实 Go2。
```

具身导航的上层逻辑可以先写成方案：

```text
自然语言目标 -> 场景识别 -> 目标点选择 -> 可通行区域 -> 路径规划 -> Go2 控制
```

后续可扩展模块：

```text
VLM / VLA 导航大脑
语义分割
动态障碍识别
目标点 grounding
MPC / PID 路径跟踪
安全停止和人工接管
```

## 12. 学习时的判断标准

你每学一个模块，都用这 5 个问题检查自己：

```text
1. 输入是什么？
2. 输出是什么？
3. 输出的坐标系是什么？
4. 输出能不能被 ROS2 标准消息表达？
5. 它怎么接到下一个模块？
```

例如 ORB-SLAM3：

```text
输入：/camera/image_raw
输出：/slam/pose、/slam/odom、/slam/map_points
坐标系：world -> camera
ROS2 消息：PoseStamped、Odometry、PointCloud2
下游模块：TF bridge、EKF、RViz、Nav2
```

例如 Go2 控制：

```text
输入：目标速度 vx / vy / yaw_rate 或目标路径点
输出：Unitree SDK2 高层控制命令
坐标系：机器人机体系或世界系，需要明确转换
ROS2 消息：Twist、Odometry、Path
下游模块：MuJoCo 仿真或真实 Go2
```

只要你能把每个模块都按这 5 个问题说清楚，这个项目就已经有了工程主线。
